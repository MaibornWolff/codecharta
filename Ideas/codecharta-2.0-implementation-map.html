Judgment calls baked in (flag if you disagree) 1. Single-lens panels go inside the lens (legend→metrics; explorer/fileExt/contextMenu→structure). 2. Genuinely cross-lens panels (metricsBar, inspector) stay in the shell and compose multiple lens facades — forcing them into one lens would re-create coupling. 3. viewCube folds into the codeMap engine, which dissolves the grandfathered codeMap↔viewCube cycle. 4. labelSettings splits: state→appearance, rendering→page. 5. attributeTypes/Descriptors split node→metrics, edge→dependency.

4 · The boundaries that pin this

Encoded as dependency-cruiser rules in the companion file Ideas/dependency-cruiser.2.0.cjs (drop-in replacement for visualization/.dependency-cruiser.js once the folders exist). The new rules, in plain words:

  • Renderer engine stays dumb — renderers/*/engine/ may not import lenses, pages, shell, interaction, appearance, or the cc.json model.
  • Lenses don't depend on UI — lenses/ may not import renderers/ or shell/.
  • Cross-lens only via the lens facade — a lens may not reach another lens's internals; external code uses <lens>.facade.ts only.
  • Only the page composes — pages import lens facades + the engine's public API; the engine never imports a page.
  • components → services → repos — feature components may not import repos/stores directly (the project already enforces the analogous rule today).
  • Only state-holders import @ngrx/store — lenses/*/repos|store, interaction/, appearance/, renderers/*/viewState/, fileStore/.
  • FileStore is the source — it imports nothing upward; the wire DTO (codeCharta.api.model) is allowed only inside fileStore/.
