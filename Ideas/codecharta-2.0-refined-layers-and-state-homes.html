CodeCharta 2.0 — Refined Layers & State Homes

Pages compose features · features read lenses (data) and write through a few state homes · every piece of state has exactly one home, chosen by its lifetime.

DECISION RECORD · ratified 2026-07-02 (naming · carved · per-renderer selection)
mapState (CodeMap renderer‑owned) sharedView (synced across renderers) preferences (durable global) lens (read-only cc.json data) fileStore (raw files)
The refined system in one sentence: a file is loaded into fileStore; each lens is a read-only projection of that cc.json; features (a top-level layer of components + services) read the lens facades they need and mutate state only through the state-home facades; pages compose features into a screen. State is not scattered — it lives in per-renderer homes (mapState, later graphState), one cross-renderer sharedView, durable preferences, read-only lenses, and the raw fileStore.

Two shifts from the current draft

1 · Features move out of lenses

The current draft nests features inside lenses (lenses/metrics/features/legend) and invents a “shell” category for cross-lens UI (metrics bar, inspector). That boundary is already a fiction — the legend “inside” the metrics lens injects the store for seven non-metrics reads. In the refined model there is one flat features/ layer; “a feature composes several lens facades + state” is the normal case, and the shell special-case disappears.

2 · State gets real homes; “lens” is reclaimed

Today state is scattered across ~30 per-setting stores under appSettings / dynamicSettings / appStatus / fileSettings. The refined model gives every slice one home, carved by lifetime (persistence × cross-renderer-sync × owning subsystem). A lens is read-only data from the file — never a home for mutable map state; the CodeMap's mutable home is mapState.

The litmus test — which home does a slice go in? (apply top-down, first match wins)
  1. Read-only projection of the immutable cc.json? → lens
  2. The raw file / file selection / delta mode? → fileStore
  3. Persisted to localStorage, true regardless of file or renderer? → preferences
  4. The same value across every renderer (focus, selection, search, filters)? → sharedView
  5. Otherwise — owned by one renderer's presentation/selection → that renderer's state (mapState, graphState)

Read facade vs write facade — where CQRS actually pays off

On the state homes, not the data lenses. A metrics/dependency lens is read-only (its only “write” is file-load seeding), so a write facade there is near-empty — do not let a lens own selection, that would turn “data from the file” into “mutable view-config owner” and break the file-load = data · user-click = view state rule. The read/write split belongs on mapState / sharedView / preferences: today appearance.facade is a single export * barrel that mixes selectors (read) and action creators (write). Splitting them lets dep-cruiser guarantee a display-only component physically cannot dispatch, and collapses the duplicated per-feature store wrappers the migration is already chasing.

Lenses stay read-only and parameterized. The metrics lens exposes rangeOf(metric) — the map feature supplies the selected metric from mapState; the lens never reads a selection itself. This dissolves the deferred valueOf cycle and unblocks the metrics-lens-ngrx-guard → error flip.

Sequence toward the end state

The first step makes state's location real at runtime (today it's folder-only). This is the store-key reshape deferred in Slice 4.

  1. Keystone — give appearance's slices their own reducer; stop combining them under appSettings → creates the real state.mapState.* root. Pure structural, but touches _applyPartialState / URL / scenario / IndexedDB paths, so it's the careful one.
  2. Move colorMode · colorRange · margin · layoutAlgorithm + the map's metric selection (area/height/color/distribution/edge) out of dynamicSettings, and hover / right-click / selectedBuilding / isLoadingMap out of appStatus/appSettings, into mapState.
  3. Stand up sharedView; move focusedNodePath · searchPattern · blacklist · markedPackages + the selected-node id in.
  4. Dissolve fileSettings: source (edges / attributeTypes / descriptors) → lenses; blacklist / markedPackages → sharedView.
  5. Purge appSettings → preferences.
  6. Genuine code refactors, last: the structure lens (also unblocks valueOf), re-expressing selectedBuildingId as a renderer-agnostic node id, and standing up graphState when the Graph renderer is built.

Kept, but deferred

A named renderer-engine seam — build the dumb-engine wrapper only when renderer #2 lands. Making the 3D map “just a feature” is what keeps the model flat, but swappable renderers are the stated point of 2.0 — and the ratified split confirms they're genuinely different: the Map shows node metrics (complexity, rloc) as a city; the Graph is a Structure101‑style LSM (levelized structure map) of dependencies between components/files. Keep the engine boundary as a named contract (load · highlight · settings + onSelect · onHover) so it can't silently rot; don't pay for the full dumb-engine/page-wire split against a single renderer that can't validate it.

Ratified decisions (2026-07-02)

a · Naming. mapState (the CodeMap renderer's own presentation + selection) · sharedView (synced across renderers) · preferences (durable global). Each future renderer owns a sibling <renderer>State (e.g. graphState).
b · Carved. Keep sharedView as its own module — the upcoming migration stories are exactly this state-home split, and multi-renderer is a real commitment.
c · Selection is per-renderer, not global. Map and Graph are fundamentally different renderers (metrics‑city vs dependency‑LSM), so they select different things. The map's metric selection lives in mapState; the Graph's in graphState. sharedView holds only cross-renderer state (focus, selection, search, filters). Open sub-note: edgeMetric is the one candidate for promotion to sharedView if a dependency-metric choice should sync between the Map's edge previews and the Graph — decide when the Graph lands.
