CodeCharta 2.0 — Lenses × Renderers

One mental model: the app is a small set of data lenses (signals from cc.json), shown through interchangeable renderers, glued by a thin layer of shared interaction state — all keyed by the stable node id.

1 · The big picture

Read it bottom-up: a file is uploaded once into a raw FileStore; each lens is a module that projects its slice of that data and exposes it through one facade (and contains its own features); a page composes the lens facades it needs into a render-model and drives a dumb renderer in the Workbench; a shared interaction layer carries what must survive a renderer switch (selection, focused node).

2 · The three axes (keep them separate)

The single most important rule. Today's pain comes from mixing these; the architecture works only if each stays distinct.

Data Lenses

Read-only signals from the file. Changes when you load a different file.

  • structure (the file tree)
  • metrics (numbers by node id)
  • dependency (edges)
  • domain / security (reserved)
  • terms (new — for WordCloud)

Output Renderers

How data is shown. Each composes several lens facades + owns its own config.

  • CodeMap, Graph, WordCloud, Report
  • one active in the Workbench
  • per-renderer view config: camera, channel mapping, layout — resets with the renderer

Interaction Shared state

User choices that must survive a renderer switch. The missing box on the first sketch.

  • focusedNodeId (the jump target)
  • file selection + delta mode
  • hovered node
Why interaction splits in two The shared part (selection, focused node) lives once and every renderer honors it — that's what makes the Graph→CodeMap jump possible. The per-renderer part (CodeMap's colorMetric/camera, Graph's layout) is owned by the renderer and resets when you leave it. Never let per-renderer config leak into the shared layer.

3 · Anatomy of one lens module

A lens is a self-contained module = a Lens Facade + several Features + lens-level Repos + a Store. Each feature = facade · models · components · services: components take data from services, and services hold logic and read the repo. This is what replaces today's 99 scattered store-facades — one facade per lens, and the feature's *Service classes become real logic instead of passthroughs.

features/lenses/metrics/
├── metricsLens.facade.ts          // the lens's public API — pages/renderers call this
├── features/
│   ├── legend/                      // a feature = facade · models · components · services
│   │   ├── legend.facade.ts
│   │   ├── legend.models.ts
│   │   ├── components/              // UI — take their data from the services
│   │   └── services/               // logic — read the repo, build the view-model
│   └── metricsBar/  …               // same shape (facade · models · components · services)
├── repos/                           // lens-level data access, shared by the features
│   ├── attributes.repo.ts          // rangeOf(metric) -> {values,min,max}, byId(id,metric)
│   └── descriptors.repo.ts         // title/description for a metric
└── store/
    └── metricsLens.store.ts         // projects lenses.metrics from FileStore, merged for visible files
The store is selection-aware FileStore holds all loaded files; the lens store projects + merges only the visible selection. So the old accumulatedDataSelector + NodeDecorator + min/max logic becomes "the lens merges its visible files" — and the analysis side already models that merge (MetricsLens.merge() / DependencyLens.merge()).

4 · Anatomy of a renderer + its page

The renderer is a dumb engine; the page is the wire that composes lens facades + view-state into a render-model and drives it. The Legend is not the renderer's panel — it's a feature inside the Metrics lens that the page mounts.

features/renderers/codeMap/
├── engine/
│   └── codeMap.renderer.ts      // DUMB engine — load(model) · highlight(id) · settings() ▸ onSelect/onHover
│                                //   (≈ today's CodeMapRenderService + ThreeScene/Three* services)
└── page/
    ├── codeMap.page.ts          // THE WIRE — composes lens facades + view-state ▸ render-model ▸ engine
    ├── codeMapViewState.store.ts   // PER-RENDERER view state: metric mapping, colorRange, camera
    ├── reads →  Structure / Metrics / Dependency lens facades   // data
    ├── reads →  SharedInteraction                               // focusedNodeId, selection
    └── mounts → renderer-scoped feature panels                  // e.g. Legend (it lives in the Metrics lens)

⚠ WordCloud needs a lens that doesn't exist yet cc.json carries structure + metrics + edges + contentHash — no source text, no identifiers, and analysis emits nothing token-like today. So WordCloud is gated on a new terms lens produced by the analysis UnifiedParser (TreeSitter). This is the architecture working as intended: the answer to "I need data X" is "add a lens," not "hack the renderer."

6 · The key that ties it all together: the node id

The cc.json 2.0 id = sha-256(canonical path), 16 hex The same id keys the structure tree, the metrics-by-id, and the edge endpoints. That single fact makes two things possible: (1) lenses join to the same tree without path guessing, and (2) "focus a1b2c3…" resolves in any renderer. The analysis 2.0 work is precisely what enables this viz architecture — the two halves are coherent.

7 · Worked example — the Legend (a CodeMap panel)

The legend reads metrics-lens data + CodeMap's view choices. Watch the three passthrough layers and the bespoke selectedColorMetricData disappear.

Today

// selector → Legend*Store → Legend*Service → component
// 9 stores + 9 services in the legend alone
class LegendColorRowComponent {
  constructor(
    colorMetricService,            // ⟶ store ⟶ selector
    colorRangeService,
    selectedColorMetricDataService,
    mapColorsService,
  ){}
  colorMetric = toSignal(colorMetricService.colorMetric$())
  colorRange  = toSignal(colorRangeService.colorRange$())
  nodeRange   = toSignal(
    selectedColorMetricDataService
      .selectedColorMetricData$())   // bespoke per-channel
  color = computed(...)
}

Lenses × Renderers

// data from a lens facade; choices from this renderer
// legend-specific state classes: 18 → 0
class LegendColorRow {
  metrics = inject(MetricsLens)        // shared facade
  codeMap = inject(CodeMapViewState)  // per-renderer

  mapColorFor = input.required()
  colorMetric = codeMap.colorMetric
  colorRange  = codeMap.colorRange
  nodeRange   = computed(() =>
    metrics.rangeOf(colorMetric()))  // generic!
  color = computed(() =>
    codeMap.mapColors()[mapColorFor()])
}

selectedColorMetricData was never state — it's just metrics.rangeOf(colorMetric()). The same generic rangeOf serves area, height, and color, so a whole family of per-channel selectors collapses.

8 · The cross-renderer jump (Graph → CodeMap)

This feature is the proof that the shared interaction layer must exist.

  1. User right-clicks a node in Graph → “Open in CodeMap”.
  2. Graph calls SharedInteraction.focusNode(id) — id is the 2.0 node id.
  3. Workbench switches the active renderer to CodeMap (Graph's camera/layout is dropped; the shared focus is kept).
  4. CodeMap mounts, reads focusedNodeId, asks StructureLens to locate it, zooms/focuses.
  5. Its legend reads MetricsLens.rangeOf(codeMap.colorMetric()) — same data, new presentation.

9 · Suggested sequencing

You have one renderer today. One renderer can't validate a renderer abstraction — two can. Build the seam on what exists, then prove it with something different.

STEP 1 · PROVE THE LENS SEAM
CodeMap as a renderer module
Carve out structure/metrics/dependency lens facades + SharedInteraction + CodeMapViewState. Legend is the first slice; delete its 18 classes.
STEP 2 · PROVE THE RENDERER SEAM
Graph (dependency lens)
Data largely exists (DependaCharta). Build the Graph→CodeMap jump → exercises the shared layer with a very different renderer.
STEP 3 · COMPOSITE
Report page
Own route, composes lens facades read-only. Must stay a consumer, never re-derive.
STEP 4 · BLOCKED ON ANALYSIS
WordCloud
Needs a new terms lens emitted by analysis. Last, gated on that story.

10 · Open questions

  • View-state register: for every current setting, decide shared vs per-renderer (selection/focus = shared; camera/channel-mapping = per-renderer). Draft the full list.
  • Terms lens: who owns the analysis story, and when? WordCloud can't ship before it.
  • Report scope: what audits/aggregations — and confirm it only composes lenses.
  • Renderer switching: exactly what persists (selection, focus) vs resets (camera, layout).
  • Migration path: the in-flight 99-facade rollout (≈55 added this month) — stop & redirect to lenses now, or finish first? Recommended: let Step 1 (CodeMap) be the redirect.
